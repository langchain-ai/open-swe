03 Phase 1 — One-Time Restructure ✓ Done 2026-06-29

Run this once. It does not touch working Daytona code — it builds the missing layers around it.

Step 1 — Rename local/self-hosted → poc/daytona

The working branch becomes the Daytona PoC reference. Rename locally and on origin.

# On local machine
git branch -m local/self-hosted poc/daytona

# Push new name to origin
git push origin poc/daytona

# Optionally delete old name on origin (kept intentionally in this setup)
# git push origin --delete local/self-hosted

# Tag the current stable state before doing anything else
git tag poc/daytona-v1.0 poc/daytona
git push origin poc/daytona-v1.0
Why tag first
The tag is immutable. Even if something goes wrong in later steps, poc/daytona-v1.0 is a permanent, safe restore point to demo from.
Note — local/self-hosted on origin
local/self-hosted was intentionally kept on origin as a historical reference. It is identical to poc/daytona at the point of rename. Do not commit to it.

Step 2 — Create base/core-custom

base/core-custom holds only the shared changes that every PoC needs. Since poc/daytona bundles shared + Daytona-specific together, the cleanest approach is to apply the shared changes fresh from main rather than trying to split the existing commit.

Step 3 — Set up Git Worktrees for Parallel Development

Git worktrees let you have multiple branches checked out simultaneously in separate directories — no git stash, no branch switching, no lost context when jumping between PoCs.

# From the main repo directory (currently on poc/daytona)
cd ~/workspace/open-swe

# Add base worktree — poc/k8s-sandbox worktree is added in Phase 2
git worktree add ../open-swe-base base/core-custom

# List all worktrees
git worktree list
Gotcha — do not create poc/k8s-sandbox from inside a worktree
If you run git checkout -b poc/k8s-sandbox from within open-swe-base/, git moves that worktree onto the new branch. Then git worktree add ../open-swe-k8s poc/k8s-sandbox fails with "already used by worktree". Fix: switch open-swe-base back to base/core-custom first, then add the worktree from open-swe/. The K8s branch creation is handled correctly in Phase 2 below.

05 Upstream Sync — The Cascade

Run this weekly (or whenever upstream merges something significant). The key rule: always rebase PoC branches onto base/core-custom, not onto main directly — otherwise you lose the shared infra changes and have to re-apply them manually.

Prerequisite — upstream remote
Before running any sync, verify the upstream remote is configured:

git remote -v → should show upstream → https://github.com/langchain-ai/open-swe.git

If it's missing, add it once:
git remote add upstream https://github.com/langchain-ai/open-swe.git 
━━━ Step 1: Fetch + impact check ━━━━━━━━━━━━━━━━━━━━
# Run from the main repo directory
cd ~/workspace/open-swe
git fetch upstream

# What files did upstream change since our last sync?
git log main..upstream/main --name-only --format="" | sort -u > /tmp/upstream-files.txt

# What files does our base layer own (differ from main)?
git diff main...base/core-custom --name-only | sort -u > /tmp/base-files.txt

# Print the intersection — files upstream touched that WE also modified.
# These are the files that will conflict. Review before touching anything.
comm -12 /tmp/upstream-files.txt /tmp/base-files.txt

━━━ Step 2: Fast-forward main ━━━━━━━━━━━━━━━━━━━━━━━
# Still in ~/workspace/open-swe/
git checkout main
git merge upstream/main --ff-only    # fails if not a clean fast-forward — investigate before proceeding
git push origin main

━━━ Step 3: Rebase base layer ━━━━━━━━━━━━━━━━━━━━━━━
# Resolve ALL shared-file conflicts HERE — not in the PoC branches
cd ~/workspace/open-swe-base
git checkout base/core-custom
git rebase main
# If conflicts: edit the file, git add , git rebase --continue
# See "Conflict ownership per layer" table below for how to resolve each file
git push origin base/core-custom --force-with-lease

━━━ Step 4: Rebase each PoC onto the updated base ━━━
# K8s PoC — resolve only K8s-specific conflicts here
cd ~/workspace/open-swe-k8s
git checkout poc/k8s-sandbox
git rebase base/core-custom
git push origin poc/k8s-sandbox --force-with-lease

# Daytona PoC — resolve only Daytona-specific conflicts here
cd ~/workspace/open-swe
git checkout poc/daytona
git rebase base/core-custom
git push origin poc/daytona --force-with-lease

━━━ Step 5: Rebuild integration ━━━━━━━━━━━━━━━━━━━━━
# SKIP — integration branch does not exist yet.
# When you need it, create it first:
#   git checkout -b integration base/core-custom
#   git push origin integration
# Then on subsequent syncs:
#   git checkout integration
#   git reset --hard base/core-custom
#   git merge --no-ff poc/daytona poc/k8s-sandbox
#   git push origin integration --force-with-lease
Why rebase onto base, not main
If poc/k8s-sandbox rebased onto main directly, it would lose the shared infra changes from base/core-custom and require re-applying them manually. Rebasing onto base/core-custom means the shared changes are already baked in — the PoC branch only deals with its own delta.

Validation script — scripts/check-ownership.sh

Copy this script into scripts/check-ownership.sh and commit it to base/core-custom. Run it before every upstream sync to see which owned files upstream has touched.

#!/usr/bin/env bash
# scripts/check-ownership.sh
# Usage: bash scripts/check-ownership.sh

set -euo pipefail
BRANCHES=("base/core-custom" "poc/daytona" "poc/k8s-sandbox")

echo "━━━ Upstream changes since last sync ━━━"
git fetch upstream -q
git log main..upstream/main --name-only --format="" | sort -u

echo ""
echo "━━━ Files owned per branch ━━━"
for branch in "${BRANCHES[@]}"; do
  parent=$(git log --pretty=format:"%D" "${branch}" | grep -oE '[^ ,]+/[^ ,]+' | head -1)
  echo ""
  echo "  Branch: ${branch}"
  git diff "main...${branch}" --name-only | sort -u | sed 's/^/    /'
done

echo ""
echo "━━━ Cross-reference: upstream ∩ owned files ━━━"
git log main..upstream/main --name-only --format="" | sort -u > /tmp/up.txt
for branch in "${BRANCHES[@]}"; do
  git diff "main...${branch}" --name-only | sort -u > /tmp/br.txt
  hits=$(comm -12 /tmp/up.txt /tmp/br.txt)
  if [ -n "${hits}" ]; then
    echo "  ⚠ ${branch} — upstream touched owned files:"
    echo "${hits}" | sed 's/^/      /'
  fi
done
echo ""
echo "Done. Resolve warnings before rebasing."

08 Maintenance Checklists

When starting a new PoC

  • Ensure base/core-custom is current with main
  • Branch new PoC from base/core-custom, NOT from main or another poc/*
  • Add worktree: git worktree add ../open-swe-<name> poc/<name>
  • Push branch and set upstream: git push origin poc/<name> -u
  • Add entry to .fork/ownership.yml
  • Add branch to scripts/check-ownership.sh BRANCHES array
  • Update GitHub branch protection rules
  • Add row to customization/upstream-sync-status.md ownership table

When adding a new file to any branch

  • Add the file path to .fork/ownership.yml under the correct branch
  • Commit the ownership update in the same PR/commit as the file addition
  • If the file is shared infra → add to base/core-custom and cherry-pick/propagate to active PoC branches

Before every demo

  • Tag the current stable state: git tag poc/<name>-demo-vN
  • Create frozen demo worktree from the tag
  • Verify demo worktree starts clean (run backend, confirm no errors)
  • Confirm WIP worktree is NOT the one being demoed
  • Push the tag to origin for the record

Weekly upstream sync

  • Run bash scripts/check-ownership.sh from ~/workspace/open-swe/ and read the output — requires the script to exist first (see Section 07)
  • Fast-forward main from ~/workspace/open-swe/ and push to origin
  • Rebase base/core-custom from ~/workspace/open-swe-base/ — resolve shared-file conflicts here, not in PoC branches
  • Rebase poc/k8s-sandbox from ~/workspace/open-swe-k8s/ onto updated base/core-custom
  • Rebase poc/daytona from ~/workspace/open-swe/ onto updated base/core-custom
  • Force-push all rebased branches with --force-with-lease
  • Verify exa-py is absent from pyproject.toml (silently re-added by upstream in the past)
  • Update customization/upstream-sync-status.md

10 The Product Journey

The PoC phase ends with a decision. Here is the full arc from PoC to product, and when to extract into a separate repo.

P1
Now → Decision
Parallel PoC Development
Multiple poc/* branches live on the fork. Each PoC is periodically rebased onto base/core-custom. Management sees frozen demo tags. Engineers develop in worktrees without interfering with each other.
P2
PoC Complete
Winning PoC Selected
Management picks the winning sandbox (say K8s). Tag it: poc/k8s-stable-v1.0. Archive or freeze losing PoC branches — don't delete, they may hold learnings. 
git tag poc/k8s-stable-v1.0 poc/k8s-sandbox
git push origin poc/k8s-stable-v1.0

# Archive losing branches (rename, not delete)
git branch -m poc/daytona archived/poc/daytona
git push origin archived/poc/daytona
git push origin --delete poc/daytona
P3
Post-Decision
Create product/main
Branch product/main from the winning PoC tag. This becomes the source of truth for the product. It continues to track upstream through the cascade. Apply branch protection and require PRs from this point. 
git checkout -b product/main poc/k8s-stable-v1.0
git push origin product/main -u

# From now on: work → PR → code review → merge into product/main
# Upstream sync still flows: main → base/core-custom → product/main
P4
Product Release
Semver Tags on product/main
Ship with product/v0.1.0, product/v0.2.0, etc. Upstream sync continues — every few weeks rebase base/core-custom onto new upstream, then product/main onto base/core-custom.
P5
Optional: High Divergence
Extract to a Separate Repository

Extract to a separate repo when: (a) the product's CI/CD, team access, and deploy pipeline diverge enough that the fork is friction, not help; (b) upstream sync conflicts are taking more than 2 hours per week; (c) you have your own release cadence independent of upstream.

If you extract: add the upstream fork as a git remote in the new repo and keep the cascade sync. The branching structure stays the same — only the GitHub org/repo changes.

Decision rule — one repo vs separate
Stay in one repo as long as upstream sync conflicts take under 1 hour per week and the product isn't deploying independently of the PoC branches. Extract when the fork becomes a liability, not before.
