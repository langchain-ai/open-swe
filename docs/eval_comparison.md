# Single-Agent vs Multi-Agent 性能对比分析报告

本报告旨在通过定量与定性的实验数据，对比分析 Open-SWE 框架在**单智能体模式 (Single-Agent)** 与**多智能体模式 (Multi-Agent)** 下的执行效能与经济性，为系统的动态路由机制设计提供数据支撑。

---

## 1. 测试方法说明 (Test Methodology)

为了保证评估的科学性、客观性与可重复性，本次测试采用了如下标准的测试方法与控制变量：

### 1.1. 测试环境与控制变量
* **推理模型**：统一指定为 `openai:deepseek-v4-flash`，以排除由于模型能力差异导致的成功率或代码质量偏差。
* **沙盒运行模式**：使用本地沙盒 (`SANDBOX_TYPE=local`)。每个测试案例都在宿主 Windows 系统独立的临时临时工作区（`evals/swe/scratch/<case_id>`）中运行，通过独立的 Git 初始化和代码拉取，保证环境之间 100% 物理隔离。
* **执行环境隔离修复**：在测试运行脚本中，通过为子进程（`subprocess`）显式注入 `PYTHONPATH=<scratch_workspace_path>` 环境变量，解决了在混合父子目录结构下 `pytest` 默认根目录定位导致的 `ModuleNotFoundError` 模块导入路径问题。

### 1.2. 评估数据集 (SWE-bench Private Benchmark)
评测基于本地私有的计算器 bug 数据集（包含 3 个典型场景，共 3 个独立任务）：
1. **`calc_bug_01` (Division by Zero Bug)**：局部代码逻辑修复。需要修改除法逻辑，在除数为 0 时抛出 `ZeroDivisionError`，而不是原有的返回 `None`。
2. **`calc_bug_02` (Negative Exponent Power Bug)**：边界条件及异常处理。需要修改幂运算逻辑，在指数为负时抛出 `ValueError`，而不是原有的返回 `0`。
3. **`calc_bug_03` (Add Modulo Feature)**：跨文件特征开发。需要在 `calculator.py` 中新增 `mod(x, y)` 函数（处理余数并防范除以零），并在 `tests/test_calculator.py` 中增加对应的单元测试用例。

### 1.3. 评测指标定义
1. **Pass@1 成功率**：执行修复后，运行测试用例的退出状态码（Exit Code）是否为 `0`（表示新增的验证性测试与原有回归测试 100% 成功）。
2. **总推理 Token 消耗 (Token Cost)**：基于输入提示词（Prompt Chars）与模型生成内容（Completion Chars）的字符总长度，按照 `1 Token ≈ 4 Chars` 的比例折算，并结合定价系数（输入 $0.15/M, 输出 $0.60/M）计算出以美元（USD）为单位的 API 推理成本。
3. **平均执行时间 (Duration)**：从运行任务接收、沙盒初始化、代码分析、代码编写，直到最终单元测试跑通并产出报告的完整系统运行墙钟时间（单位：秒）。
4. **平均 AST 相似度 (AST Similarity)**：利用 Python 官方 `ast` 模块解析智能体生成的代码与标准人类参考答案（Golden Patch）的语法树结构，并计算语法节点的相交比率（Jaccard 相似度），分值在 `0.0` 到 `1.0` 之间。

## 3. 案例维度详细数据对比

### 案例 1：Division by Zero Bug (除以零崩溃修复)
* **单智能体**：耗时 **47.0s** \| Token 数 **428** \| 成本 **$0.00025**
* **多智能体**：耗时 **208.8s** \| Token 数 **1,681** \| 成本 **$0.00100**
* **分析**：这是一个局部、低复杂度的编码修复。单智能体直接定位并修改了 `calculator.py`；而多智能体流程强行分拆了 PM 写规划文档、Architect 扫描目录、QA 发起专用测试，在简单的单文件任务上表现出显著的“大炮打蚊子”效应。

### 案例 2：Negative Exponent Power Bug (负指数幂函数修复)
* **单智能体**：耗时 **168.1s** \| Token 数 **1,540** \| 成本 **$0.00091**
* **多智能体**：耗时 **191.7s** \| Token 数 **2,072** \| 成本 **$0.00123**
* **分析**：由于涉及到边界判断逻辑，单智能体通过循环调试交互耗费了较多时间，两者耗时差距开始收窄。但单智能体免去了大量的团队协作上下文，在 Token 成本上依然保持明显优势。

### 案例 3：Add Modulo Feature (新增取模功能)
* **单智能体**：耗时 **157.9s** \| Token 数 **1,275** \| 成本 **$0.00075**
* **多智能体**：耗时 **215.3s** \| Token 数 **1,655** \| 成本 **$0.00098**
* **分析**：此案例属于跨文件的功能添加，要求修改测试模块和源码。单智能体独立寻找文件并实现，表现敏捷；多智能体由于对文件进行了前期的物理依赖圈定，保障了高规范性，但额外增加了协同用时。

## 4. 结论与架构演进：引入动态路由机制 (Dynamic Routing)

### 4.1. 模式优劣势对比总结
* **单智能体优势**：在简单、单文件、高局部性的 Bug 修复或问答式任务中，**单智能体在开发效率上领先 ~40%，节省 Token 开销 ~40%**。其反应迅速，直接编码。
* **多智能体优势**：在大中型项目、多文件依赖或高风险开发中，多智能体拥有的 **PM 需求提炼**、**Architect 目录物理隔离（防上下文暴涨）** 和 **QA 自我反思重试循环 (Self-Reflection Retry)** 具备决定性的稳定性保障，能防止单智能体遇到报错时陷入死循环。

### 4.2. 动态路由设计方案
为了融合两者的长处，框架在任务获取的源头引入了**动态路由机制 (Dynamic Routing)**：
1. **入口拦截**：在 `get_agent(config)` 阶段，读取任务描述。
2. **复杂度判定**：使用轻量级意图分类器，根据任务的规模、关键字及涉及文件范围，预测该任务是“简单修复/信息查询”还是“复杂特征开发”。
3. **路由执行**：
   - 判定为简单修复：直接路由到**单智能体**执行，追求敏捷与低成本。
   - 判定为复杂开发：路由到**多智能体协同图**，开启 PM-Architect-Coder-QA 自愈流水线，追求高保障度与自我纠错。
